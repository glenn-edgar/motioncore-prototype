## 1. What you're building

A site-wide catalog of dongle classes (analogous to the existing `robot_classes.lua` catalog). Each class is identified by a string name; its FNV-1a 32-bit hash becomes the wire-level `class_id`. A dongle's firmware bakes its own `class_id` constant at build time; a robot's container loads the catalog at runtime and matches REGISTER frames against its expected-dongles list.

Your work has four pieces:

1. **New catalog file** — `catalogs/dongle_classes.lua`
2. **kb_build codegen extension** — emit `class_ids.{h,lua,json}` from the catalog
3. **Robot-class catalog extension** — `required_dongles` field
4. **Per-robot instance config extension** — `dongles[]` array
5. **Robot-side runtime matching** — chain-tree subtree picks up REGISTER frames

## 2. File-by-file specification

### 2.1 New catalog file

**Path:** `nano_data_center_base/commissioning_software/system_node_control/construction/catalogs/dongle_classes.lua`

**Format:** identical pattern to the sibling `robot_classes.lua` catalog already in this directory.

```lua
-- =============================================================================
-- catalogs/dongle_classes.lua
--
-- Site-wide dongle CLASS catalog. One entry per dongle class. A class is
-- identified on the wire by its FNV-1a 32-bit hash, computed at codegen
-- time from the entry's key string. Bumping the suffix (_v1 -> _v2) is
-- a deliberate compatibility break — different hash, different class.
--
-- Per-entry shape:
--   description    string   human-readable docs
--   capabilities   list?    optional canonical capability tags
--                          (HIL functional, HIL analytic, router, etc.)
--   primary_role   string?  human label of the primary deployment role
--
-- Empty fixture is valid — sites without any dongle classes still build.
-- =============================================================================

return {

    capabilities = { "gpio", "pwm", "adc_12bit", "dac_10bit", "counter", "quadrature" },
```

### 2.2 kb_build codegen extension

**What kb_build currently does for robot_classes.lua:** look at how that catalog is consumed by the build pipeline (likely emits .h / .lua / .json artifacts and stages them into containers). Mirror that for `dongle_classes.lua`.

**Three outputs required:**

#### `class_ids.h`

For dongle firmware builds (consumed by `motioncore-prototype/samd21/.../Makefile`, etc.).

```c
// Generated by kb_build. DO NOT EDIT.
// Source: catalogs/dongle_classes.lua
#ifndef DONGLE_CLASS_IDS_H
#define DONGLE_CLASS_IDS_H

#include <stdint.h>

#define CLASS_SAMD21_SHELL_V1     ((uint32_t)0xXXXXXXXX)  // fnv1a("samd21_shell_v1")
#define CLASS_RA4M1_ANALYTIC_V1   ((uint32_t)0xXXXXXXXX)
#define CLASS_RP2350_ROUTER_V1    ((uint32_t)0xXXXXXXXX)
#define CLASS_ESP32C6_ROUTER_V1   ((uint32_t)0xXXXXXXXX)
#define CLASS_PC_INTERNAL_V1      ((uint32_t)0xXXXXXXXX)

#endif // DONGLE_CLASS_IDS_H
```

The XXX values are the actual FNV-1a 32-bit hashes computed from each catalog key string.

#### `class_ids.lua`

For Lua/host-side consumers — provides both name→hash and hash→name lookups.

```lua
-- Generated by kb_build. DO NOT EDIT.
-- Source: catalogs/dongle_classes.lua
local M = {
  by_name = {
    samd21_shell_v1     = 0xXXXXXXXX,
    ra4m1_analytic_v1   = 0xXXXXXXXX,
    rp2350_router_v1    = 0xXXXXXXXX,
    esp32c6_router_v1   = 0xXXXXXXXX,
    pc_internal_v1      = 0xXXXXXXXX,
  },
  by_hash = {
    [0xXXXXXXXX] = "samd21_shell_v1",
    [0xXXXXXXXX] = "ra4m1_analytic_v1",
    [0xXXXXXXXX] = "rp2350_router_v1",
    [0xXXXXXXXX] = "esp32c6_router_v1",
    [0xXXXXXXXX] = "pc_internal_v1",
  },
}

function M.resolve(name)   return M.by_name[name] end
function M.name_of(hash)   return M.by_hash[hash] end

return M
```

#### `class_ids.json`

For documentation, diagnostic UI, tooling.

```json
{
  "samd21_shell_v1":   "0xXXXXXXXX",
  "ra4m1_analytic_v1": "0xXXXXXXXX",
  "rp2350_router_v1":  "0xXXXXXXXX",
  "esp32c6_router_v1": "0xXXXXXXXX",
  "pc_internal_v1":    "0xXXXXXXXX"
}
```

### 2.3 FNV-1a 32-bit hash function

Identical to `s_expr_hash` in `motioncore-prototype/s_engine/runtime/s_engine_types.h`. Reference implementation:

```c
#define FNV_OFFSET  0x811C9DC5u
#define FNV_PRIME   0x01000193u

static uint32_t fnv1a_32(const char* str) {
    uint32_t hash = FNV_OFFSET;
    while (*str) {
        hash ^= (uint8_t)*str++;
        hash *= FNV_PRIME;
    }
    return hash;
}
```

Lua reference (from `chain_tree_luajit` or equivalent):

```lua
local function fnv1a_32(str)
  local hash = 0x811C9DC5
  for i = 1, #str do
    hash = bit.bxor(hash, str:byte(i))
    hash = bit.band(hash * 0x01000193, 0xFFFFFFFF)
  end
  return hash
end
```

**Test vector** (same as canonical):
```
fnv1a_32("123456789") == 0xDF85ED5A   -- NOTE: this is the 32-bit equivalent;
                                       -- crc8_autosar's 0xDF is unrelated
```

The hash MUST be byte-identical between Lua and C implementations. Run a self-test at codegen time on the string `"123456789"` to verify both implementations agree.

### 2.4 Robot-class catalog extension

The existing `catalogs/robot_classes.lua` per-entry shape gains a `required_dongles` list. Update the docstring at the top of that file:

```lua
return {
  lunar_rover_v2 = {
    description      = "Lunar rover with door + LED + IMU",
    capabilities     = { "recharge", "dock_in", "dock_out", ... },
    required_dongles = {
      -- Class names from catalogs/dongle_classes.lua. Order is convention only.
      "samd21_shell_v1",      -- door
      "samd21_shell_v1",      -- LED panel
      "ra4m1_analytic_v1",    -- current loop
      "rp2350_router_v1",     -- IMU CAN bus
    },
  },
}
```

The catalog validation should: for each `required_dongles` entry, verify the class name exists in `dongle_classes.lua`. Fail kb_build with a clear error if not.

### 2.5 Per-robot instance config extension

Existing instance configs (e.g., `containers/lunar_rover-class/stage/rover_1_config.json` or wherever they live in the build pipeline) gain a `dongles` array:

```json
{
  "robot_id":    "rover_1",
  "robot_class": "lunar_rover_v2",
  ...
  "dongles": [
    { "role": "door_control",  "class": "samd21_shell_v1",   "instance": 1 },
    { "role": "led_panel",     "class": "samd21_shell_v1",   "instance": null },
    { "role": "current_loop",  "class": "ra4m1_analytic_v1", "instance": 257 },
    { "role": "imu_bus",       "class": "rp2350_router_v1",  "instance": null }
  ]
}
```

**Field semantics:**

| Field | Type | Meaning |
|---|---|---|
| `role` | string | Free-text role label, used by the robot's task layer to find this dongle |
| `class` | string | Class name from `dongle_classes.lua`. Resolved to `class_id` hash at robot startup. |
| `instance` | uint32 \| null | `null` = no-id-match (any dongle of this class binds to this role, first match wins). Integer 1..0xFFFFFFFF = specific commissioned instance must match exactly. |

**Build-time validation:**
- Every `class` value must exist in `dongle_classes.lua`
- Every robot's `dongles[]` must satisfy its class's `required_dongles[]` (every required class must appear in `dongles[]` at least once)
- Optional warn: roles repeated within a single robot's config

### 2.6 Robot-side runtime matching

The robot's chain-tree subtree (probably extending `ct_comm.lua` and the comm-tx/comm-rx columns in `ros_planner_ii_mqtt_robot/`) needs to:

#### On startup:
1. Read the instance config's `dongles[]` array
2. Load `class_ids.lua` registry
3. Resolve each `class` name to its `class_id` hash
4. Build an in-memory lookup table:
   ```lua
   -- (class_id, instance_id or "any") -> { role = "...", handler_state = "looking" }
   ```
5. Build a "not-mine dedup set" — empty initially

#### On every received REGISTER frame:
```
register = { class_id, instance_id, commissioning_state, chip_uid, vid, pid, fw_version, build_date }

-- Try exact match first
slot = expected_dongles[(register.class_id, register.instance_id)]

-- Fall back to any-instance match
if not slot then
  slot = expected_dongles[(register.class_id, "any")]
end

if slot and slot.handler_state == "looking" then
  -- Match: bind dongle to role
  slot.handler_state = "bound"
  slot.bound_dongle = register
  log("dongle bound: role=" .. slot.role
      .. " class=" .. class_ids.name_of(register.class_id)
      .. " instance=" .. register.instance_id
      .. " chip_uid=" .. hex(register.chip_uid))
  -- Send OP_REGISTER_ACK
  comm_submit(...)
elseif slot then
  -- Already bound — this is a heartbeat-replacement, fine, ignore silently
elseif not_mine_seen[(register.class_id, register.instance_id)] then
  -- Already logged once, increment counter
  not_mine_seen[(register.class_id, register.instance_id)].count = ... + 1
else
  -- First sight of an unrecognized dongle. Log once, dedup future sights.
  not_mine_seen[(register.class_id, register.instance_id)] = { count = 1 }
  log("dongle 'not mine': class=" .. (class_ids.name_of(register.class_id) or "0x" .. hex(register.class_id))
      .. " instance=" .. register.instance_id
      .. " — not in expected_dongles, ignoring")
end
```

The **"not mine"** path is load-bearing — see the design memory for the multi-robot rationale.

## 3. ABI contract — class_ids.h

The **byte-stable interface** between this repo and `motioncore-prototype/` is `class_ids.h`. The motioncore-prototype dongle firmware builds include this file via `-I` flag and bake their class_id constant via:

```c
#include "class_ids.h"
#define MY_CLASS_ID  CLASS_SAMD21_SHELL_V1   // or whichever class this firmware targets
```

Stability rules:
- Hashes are computed from the catalog string. The string IS the contract.
- Adding a new class never changes existing hashes — safe.
- Renaming an existing class (without the _vN bump) is FORBIDDEN — would silently break deployed firmware.
- Bumping _v1 → _v2 is the designated way to break compatibility — kb_build should warn when an existing class disappears from the catalog (deletion is allowed but it's worth a "did you mean to bump _v2?" prompt).

## 5. Verification

After implementing 2.1-2.5:

1. **Hash test vector**: in kb_build, compute `fnv1a_32("123456789")` in Lua, verify it equals the C implementation's result (use a tiny C test program). Both should produce the same uint32.

2. **Catalog round-trip**: emit `class_ids.lua`, load it, assert that `M.by_name[name] == M.by_hash` reverse-lookup for every entry.

3. **Cross-reference**: validate that every `required_dongles` entry in `robot_classes.lua` exists as a key in `dongle_classes.lua`. Fail build cleanly if not.

4. **Instance config**: validate that every robot's `dongles[].class` exists in `dongle_classes.lua` AND that every class in the robot's `required_dongles` is represented at least once in the instance's `dongles[]`.

5. **Smoke test runtime**: write a small Lua test that fakes a REGISTER frame and runs it through the matching code path. Verify role binding works for exact-match, no-id-match (`instance: null`), and not-mine.

6. **Documentation update**: extend the existing kb_build user docs to mention `dongle_classes.lua` alongside `robot_classes.lua`.
